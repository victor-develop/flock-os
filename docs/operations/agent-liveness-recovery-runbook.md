# Agent Liveness Recovery Runbook

Owner: Software Architect · Supersedes the CEO-only runbook ([FLO-267](/FLO/issues/FLO-267)) · Mechanism issue: [FLO-395](/FLO/issues/FLO-395) · Detection library: `tools/ops/agent_liveness/`

This runbook defines the **automatic** recovery path for a stuck/silent run on **any senior agent** — not just the CEO — and the **manual** override a board operator uses when automation cannot resolve it. It is the deterministic program every **Liveness Watchdog** routine executes on every fire, and the reference a human follows during an incident.

## Why this exists

The CEO watchdog ([FLO-267](/FLO/issues/FLO-267)) proved the detect → release → restart → escalate pattern for the CEO. But it was CEO-specific: the recovery owner was a hardcoded peer (the Software Architect). On **2026-06-20** the Software Architect itself went silent on [FLO-365](/FLO/issues/FLO-365) for ~3h+. The platform's stale-run detector fired correctly ([FLO-376](/FLO/issues/FLO-376)), **but the recovery owner was the same stuck agent** — so the recovery could not self-heal while that agent's run slot was occupied. The org's #2 agent was effectively offline, and two issues queued on it ([FLO-376](/FLO/issues/FLO-376), [FLO-382](/FLO/issues/FLO-382)) were stranded.

The fix is one rule, applied to the whole chain of command:

> **Recovery owner = the stuck agent's manager.** Never the stuck agent itself.

That is the core of this runbook. The CEO remains covered (its manager is the board; the configured peer — Software Architect — owns the release+restart attempts and the board is the escalation backstop), and every other senior agent gains the same self-heal.

## Architecture (defense in depth)

```
                ┌──────────────────────────────────────────────┐
  any agent     │  assignment / heartbeat / scheduled trigger   │
  run           └──────────────────────────────────────────────┘
                              │ liveness signal (staleness)
                              ▼
   ┌────────────────────────────────────────────────────────────┐
   │  <Agent> Liveness Watchdog  (owner: stuck agent's MANAGER)  │
   │  triggers: schedule (backstop) + webhook (event path)       │
   │                                                             │
   │   detect  →  terminate(stuck slot)  →  restart(backoff ×3)  │
   │                                          │ fail ×3           │
   │                                          ▼                   │
   │                                  escalate to board           │
   └────────────────────────────────────────────────────────────┘
        ▲                                  │ integrates when ready
        │ event path (webhook)             │
        └──────── FLO-266 monitoring ──────┘   (DevOps)
   FLO-265 timeout enforcement caps run duration at the platform level
   (independent of this runbook; listed here for completeness).
```

| Layer | Issue | Owner | Role |
|-------|-------|-------|------|
| Timeout enforcement | [FLO-265](/FLO/issues/FLO-265) | DevOps | Caps a single run's max duration (platform level) |
| Monitoring & observability | [FLO-266](/FLO/issues/FLO-266) | DevOps | Emits staleness alerts; can fire the watchdog webhook |
| **Recovery (this runbook)** | [FLO-267](/FLO/issues/FLO-267) (CEO) / [FLO-395](/FLO/issues/FLO-395) (generalized) | Software Architect (CEO watchdog) / each agent's manager | Detect → terminate → restart → escalate |

This runbook is **self-contained**: each watchdog performs its own detection and recovery so it works even before FLO-265/FLO-266 are wired to it. When they are, the watchdog gains an event-driven webhook trigger (no idle polling cost) and respects the platform timeout.

> **Why the manager, not a peer.** A peer's heartbeats are independent of the stuck agent's, so a peer *can* run recovery. But a manager is the right owner: it has the authority to re-prioritise/reassign the stuck agent's work, it is the natural escalation point, and routing to the manager avoids the FLO-365 failure mode where the only available peer *is* the stuck agent. For the CEO (no manager), a configured peer is the fallback and the board is the backstop — the original FLO-267 design, preserved.

> **Routine ownership.** Paperclip only lets an agent create routines **assigned to itself**. So each manager provisions the watchdog for its own reports: the Software Architect owns the DevOps/Frontend/QA/Backend watchdogs; the **CEO** owns the Architect watchdog. Provisioning the Architect watchdog is therefore a CEO action — tracked in the follow-up issue linked from [FLO-395](/FLO/issues/FLO-395).

## Constants

| Constant | Value | Rationale |
|----------|-------|-----------|
| `T_SUSPICIOUS` | 15 min | Exceeds a normal 5–10 min run + one cycle |
| `T_STUCK` | 20 min | Detect threshold — recovery starts here |
| `T_ESCALATE` | 30 min | Hard threshold — board must be notified |
| `T_ABORT` | 45 min | Stop automated retry; human-only from here |
| `MAX_ATTEMPTS` | 3 | Release+restart attempts per recovery action |
| Backoff | 30 s → 60 s → 120 s | Exponential delays between attempts |

These are the defaults in `tools/ops/agent_liveness/recovery.py` (`LivenessThresholds`, `DEFAULT_MAX_ATTEMPTS`, `DEFAULT_BACKOFF_SECONDS`). A watchdog may override them per-trigger if a specialised agent needs a different cadence (e.g. a long-running batch agent), but the defaults match the CEO watchdog and cover the incident shape.

## Automatic recovery — what the watchdog does each fire

The watchdog is the **recovery-owner agent** waking on a **`<Agent> Liveness Watchdog`** run issue. On every fire it runs the procedure below and records the outcome as a comment on the run issue (and on the watched agent's open stale-run review issue, e.g. the platform-created *Review silent active run for …* issue, only when it acts).

### Step 1 — Detect (read liveness)

Resolve the watched agent's most recent **base run** — the most recent run whose `status` is **not** `coalesced` (its `coalescedIntoRunId` is null). For a heartbeat-routine agent (the CEO), read it from the routine run history:

```bash
GET /api/routines/{WATCHED_ROUTINE_ID}/runs?limit=20
```

For an assignment-driven agent (everyone else), the cleanest signal is the platform's stale-run detector. Search for the open review issue it created:

```bash
GET /api/companies/{companyId}/issues?q=Review+silent+active+run+for+{AgentName}
```

Filter to non-terminal statuses (`todo`/`in_progress`/`blocked`). The issue body carries the run id, started-at, last-output-at, and silent duration — everything the verdict needs. If no such issue exists, the agent is healthy → exit.

Let `linkedIssueId` = the stuck run's source issue, `startedAt` = run's `triggeredAt`, and `age` = now − `startedAt`. Fetch the linked issue to read its real status:

```bash
GET /api/issues/{linkedIssueId}
```

**Verdict** (matches `classify_run()` in the library):

- If the linked issue is terminal (`done`/`cancelled`) → **healthy**. Exit.
- If the linked issue is under an active `missing_disposition` recovery action (cause `successful_run_missing_state`) → **healthy**. Exit. (See the FLO-771 note below.)
- If `age < T_SUSPICIOUS` → **healthy**. Exit.
- If `T_SUSPICIOUS ≤ age < T_STUCK` → **suspicious**. Log a watch comment. Exit.
- If `age ≥ T_STUCK` and the linked issue is still active → **STUCK**. Proceed to Step 2.
- If `age ≥ T_ABORT` → **abort** (skip Step 3, go to Step 4).

> **Zombie run-records (FLO-419).** A strictly newer non-coalesced base run in the window supersedes an older in-flight run — under `coalesce_if_active` the platform only admits a fresh base run once the prior one is inactive, so the older record is stale, not stuck. Treat such a run as **healthy** (`is_superseded_by_newer_base_run()` in the library) and do not alert or recover it.

> **Successful run, missing disposition (FLO-771).** A run may **succeed** (the run itself completed and produced output) yet leave the linked issue without a terminal status — the platform then opens an `activeRecoveryAction` with `kind: "missing_disposition"` and `cause: "successful_run_missing_state"`, routing the issue to the recovery owner for cleanup. This is **not** a silent/stuck process; a STUCK → release+restart would interrupt a healthy agent and not fix the disposition gap. Read the linked issue's `activeRecoveryAction`; when `kind == "missing_disposition"` and `cause == "successful_run_missing_state"`, set the `successful_run_missing_disposition` flag and treat the run as **healthy** (skip Step 2). The watchdog retains judgment: if the agent was *also* genuinely silent, escalate via the manual path — this flag only suppresses automated release+restart on a run whose own success is attested. (Observed during the 2026-06-22 [FLO-769](/FLO/issues/FLO-769) CEO watchdog fire.)

> **Composability:** when FLO-266 lands, prefer its monitoring signal (or the webhook trigger) as the detection source and use the run-history / stale-run-issue check above as the fallback only.

### Step 2 — Resolve the recovery owner

```python
owner = resolve_recovery_owner(stuck_agent_id, chain, ceo_fallback_owner_id=ARCHITECT_ID)
# Architect stuck -> owner = CEO. DevOps stuck -> owner = Architect. CEO stuck -> owner = Architect (peer).
assert owner != stuck_agent_id   # the whole point of FLO-395
```

The recovery owner is **the stuck agent's manager**, resolved from `GET /api/companies/{companyId}/agents` (each agent carries `reportsTo`). Never the stuck agent. For the CEO (no manager), the configured peer (Software Architect) is the fallback owner and the board is the escalation backstop.

### Step 3 — Terminate + Restart (release stuck slot, auto-restart with backoff, max 3)

Release the stuck run's source issue so the next trigger is not coalesced/skipped away:

```bash
POST /api/issues/{linkedIssueId}/release
{ "agentId": "{RECOVERY_OWNER_ID}" }
Headers: Authorization: Bearer $PAPERCLIP_API_KEY, X-Paperclip-Run-Id: $PAPERCLIP_RUN_ID
```

> **Scope note.** Releasing frees the issue checkout/slot so a fresh run can be admitted. It does **not** kill the underlying adapter process — capping process duration is FLO-265's job (platform timeout). If `release` returns `401/403` (the watchdog lacks authority over the stuck agent's checkout), skip the retry loop and go straight to **Step 4 — Escalate**.

Then kick a fresh run. For a heartbeat-routine agent:

```bash
POST /api/agents/{WATCHED_AGENT_ID}/heartbeat/invoke
Headers: Authorization: Bearer $PAPERCLIP_API_KEY, X-Paperclip-Run-Id: $PAPERCLIP_RUN_ID
```

For an assignment-driven agent, re-admit by reassigning the source issue to the same agent (a fresh checkout wakes it) or by leaving the release to let the next scheduled retry/comment wake fire. Pseudocode (`plan_recovery()` in the library):

```
for attempt in 1..MAX_ATTEMPTS:
    if attempt > 1: sleep(backoff[attempt-1])        # 30s, 60s, 120s
    release linked issue
    invoke fresh run (heartbeat or reassign)
    sleep(45s)                                       # let it admit
    re-detect (Step 1)
    if linked issue terminal OR a NEW non-coalesced run appeared:
        mark RECOVERED on attempt N; break
if not recovered: go to Step 4
```

A recovery is **successful** when a fresh non-coalesced run appears and progresses (new output / linked issue advances), or the prior linked issue moves to terminal.

Record a single comment on the watchdog run issue (and the stale-run review issue, if any): *“Watchdog recovered `<watched agent>` run `<base run id>` after `<age>` min via release+invoke (attempt N; owner=`<manager>`).”*

### Step 4 — Escalate (automation exhausted)

Triggered when: 3 attempts fail, **or** `age ≥ T_ESCALATE` and Step 3 was denied, **or** `age ≥ T_ABORT`. Stop retrying — do not hammer the agent.

- Post an **incident comment** on the watchdog run issue + the watched agent's stale-run review issue summarising detection time, base run id, attempts made, recovery owner, and the exact error (e.g. release `403`).
- Create a **board approval** (`POST /api/companies/{companyId}/approvals`, `type: request_board_approval`) requesting authorisation to force-release / pause-resume the stuck agent, linking `issueIds: ["<stale-run review issue id>"]`.
- Do **not** auto-pause the agent without board authorisation (pausing a senior agent has org-wide blast radius).

## Manual override (board / operator)

Use when automation is paused, has failed, or during a declared incident. All calls need a board/operator API key and `X-Paperclip-Run-Id` is not required for manual ops.

### Fast path — force a fresh run

```bash
POST /api/agents/{stuck-agent-id}/heartbeat/invoke        # heartbeat-routine agent
# or, for an assignment-driven agent, reassign the stuck issue to re-wake it:
PATCH /api/issues/{stuck-issue-id} { "assigneeAgentId": "{stuck-agent-id}" }
```

### Free a stuck checkout

```bash
POST /api/issues/{stuck-issue-id}/release
```

### Pause / resume an agent (use only if restarts keep failing)

```bash
POST /api/agents/{stuck-agent-id}/pause     # stops heartbeats / scheduled wakes
POST /api/agents/{stuck-agent-id}/resume    # resumes
```

### Roll back a bad agent config

If the stuck run followed a config change, roll it back:

```bash
GET    /api/agents/{stuck-agent-id}/config-revisions
POST   /api/agents/{stuck-agent-id}/config-revisions/{revisionId}/rollback
```

## Watchdog routine configuration (per covered agent)

One routine per watched agent. The Software Architect owns the watchdogs for its reports; the CEO owns the Architect watchdog. Created **paused** and activated once the runbook + negative test are reviewed (the FLO-267 safe-launch pattern).

| Field | Value |
|-------|-------|
| Title | `<Agent> Liveness Watchdog` (e.g. `DevOps Liveness Watchdog`) |
| Assignee | The watched agent's **manager** (the recovery owner) |
| Project | Flock OS Platform (`747bf0c1-8744-4bd8-b215-2d1456dc1236`) |
| Concurrency | `coalesce_if_active` (one watchdog run at a time) |
| Catch-up | `skip_missed` |
| Initial status | **`paused`** — activates on manager/board approval |

**Triggers:**

1. **Schedule (backstop):** offset ~10 min after the watched agent's cadence. For the CEO (`*/15 * * * *`), that is `10,25,40,55 * * * *` UTC. For assignment-driven agents with no fixed cadence, a 30-min backstop (`10,40 * * * *`) is the budget-aware default — lean on the FLO-266 webhook as the primary path.
2. **Webhook (event path):** firing URL is emitted when the trigger is created; wire FLO-266 monitoring to POST here on a staleness alert for zero-idle, event-driven recovery.

> **Cadence vs budget.** A per-agent backstop cadence is justified while adapter spend is $0 (local model). If the company moves to a metered adapter, collapse to one watchdog-per-manager on a 30-min cadence and lean on the FLO-266 webhook as the primary path.

## Provisioning checklist (per new watchdog)

1. Confirm the watched agent's manager (the recovery owner) — see §Coverage.
2. Manager creates the routine assigned to **itself** (Paperclip: agents can only create routines assigned to themselves).
3. Add the schedule + webhook triggers (above).
4. Leave `status: paused`; activate after the manager reviews this runbook + the negative test (`tools/ops/agent_liveness/tests/test_recovery.py`) is green.
5. Record the routine id in the §Coverage table's runtime overlay (or a secrets/store entry) so detection can find it.

## Verification

The watchdog run is itself the verification loop: every fire either confirms healthy (no-op comment) or performs measurable recovery. The pure logic is pinned by `tools/ops/agent_liveness/tests/test_recovery.py` — including the canonical **negative test** (`test_simulated_silent_architect_run_self_heals`) that replays the FLO-365 incident shape and asserts the recovery owner resolves to the CEO (never the Architect) and that the release+restart loop escalates to the board once attempts are exhausted — the **zombie-run** test (`test_zombie_run_does_not_trigger_recovery`, FLO-419) that asserts a superseded run is detected + classified healthy so a moved-on agent raises no false alert — and the **missing-disposition** test (`test_successful_run_missing_disposition_does_not_trigger_recovery`, FLO-771) that asserts a run whose success is attested (but whose issue lacks a terminal status) classifies healthy so a producing agent is not interrupted.

To exercise the live path safely:

1. With the watchdog **paused**, manually fire it (`POST /api/routines/{routineId}/run`) while the watched agent is healthy — it should detect *healthy* and exit without acting.
2. To simulate a stuck slot without harming the agent, create a throwaway `in_progress` issue, point a dry-run detection query at it, and confirm the verdict logic (`classify_run`) produces *STUCK* at the expected age. (Do not release real issues in a drill.)
3. Assert `resolve_recovery_owner(watched_agent) != watched_agent` for every covered agent — this is the FLO-395 invariant and is pinned in the test suite.
